## 为什么需要超越 JSVMP

JavaScript VMP（虚拟机保护）自 2018 年以来一直是 Web 端代码保护的事实标准。核心思路很简单：把 JavaScript 编译成自定义字节码，然后连同字节码解释器一起下发。攻击者看到的是字节码而非源码，逆向门槛由此提高。

但 JSVMP 有一个根本性的弱点：**程序在任何时刻都以确定的、可复制的状态存在**。攻击者可以：

- 随时对内存做快照
- 设断点检查解释器内部状态
- 逐步追踪执行流程
- 对比多次运行（它们是完全相同的）

所有现有的保护方案——JSVMP、WASM 混淆、控制流平坦化、字符串加密——都共享这个假设。本文探索 6 种打破这一假设的方向。

## 共同的假设

所有传统保护方案都假设：

```
程序 = 客户端上的固定代码
攻击者可以：快照、追踪、对比、重放
每次执行 = 相同的字节码
```

本文的 6 个方向以不同方式打破这个假设：

| 方向 | 如何打破假设 |
|---|---|
| 量子混淆 | 状态是叠加态而非确定态；测量即坍缩 |
| 拓扑保护 | 代码编码为纽结拓扑，而非顺序指令 |
| OTP 流 | 每次执行代码都不同（不同的密钥流） |
| iO | 代码在数学上与随机不可区分 |
| ZK-WASM | 代码根本不在客户端——只有证明在 |
| Local-Mixing | 代码存在但被打散到无法识别 |

## 量子电路混淆

### 原理

将经典 Boolean 电路编译为量子电路。保护来自量子力学的两个基本定理：

**不可克隆定理**：物理上不可能创建一个未知量子态的完美副本。攻击者无法对程序状态做快照。

**测量坍缩**：观测量子叠加态会导致其坍缩为经典状态，破坏叠加信息。攻击者无法在不破坏计算的情况下检查中间状态。

### 编译路径

```
经典电路 (AND/OR/NOT/XOR)
  → 量子门 (Toffoli/CNOT/X/Hadamard)
  → 相位随机化 (Z/S/T/RZ 门)
  → 辅助量子比特注入 (纠缠垃圾比特)
  → 基变换 (Hadamard 基 ↔ 计算基)
  → 混淆量子电路
```

关键洞察：**相位门**（Z、S、T）只改变量子振幅的相位，不改变测量概率。插入任意 Z/S/T 门不会改变程序输出，但会完全改变态矢量。每次用不同随机种子混淆，产生的态矢量看起来完全不同。

### 数学基础

n 个量子比特的状态是 2^n 维复数向量：

```
|ψ⟩ = Σ α_i |i⟩    其中 Σ|α_i|² = 1
```

量子门是酉矩阵 U，满足 U†U = I。

相位门：

```
Z|0⟩ = |0⟩     Z|1⟩ = -|1⟩
S|0⟩ = |0⟩     S|1⟩ = i|1⟩
T|0⟩ = |0⟩     T|1⟩ = e^{iπ/4}|1⟩
```

这些门只旋转 |1⟩ 分量的相位。测量概率 |α_i|² 不变，但态矢量完全不同。

### 攻击结果

| 攻击 | 信息获取 | 结果 |
|---|---|---|
| 中间态测量 | 0% | 态坍缩，计算被破坏 |
| 不可克隆定理 | 0% | 叠加态无法复制 |
| 门序列分析 | 10% | 相位门淹没逻辑门 |
| 差分分析 | 50% | 仅黑盒 |
| 统计推断 | 40% | 确定性输出 |
| 量子态层析 | 50% | 需要 O(4^n) 次测量 |
| **平均** | **30%** | |

### V8 映射

量子模拟器以 Float64Array 矩阵运算在 JavaScript 中运行。每个门是对态矢量的一次矩阵乘法。局限是 2^n 的指数内存开销，实际仅适用于 n ≤ 20 量子比特。

### 局限

此方案主要是概念性的。真实量子硬件不存在于浏览器中，经典模拟有指数开销。其价值在于展示量子力学原理提供了一种根本不同的保护模型。

## 拓扑保护执行

### 原理

将程序逻辑编码为纽结（knot）的拓扑结构。执行 = 计算拓扑不变量（链接数）。逆向 = 解纽结等价问题，该问题属于 NP。

与量子混淆不同，这是纯经典数学。不需要硬件，硬度来自数学结构本身。

### 编译路径

```
经典电路 (AND/OR/NOT/XOR)
  → 辫群元素 (σ_i 生成元序列)
  → 闭合辫 = 纽结/链环
  → Reidemeister 变形 (3 种，保持纽结类型)
  → 垃圾分量注入 (与主纽结链接但不影响输出)
  → 混淆纽结 (拓扑等价但视觉完全不同)
```

每个逻辑门映射到一个辫字：

- **AND(a,b)**：`σ_a σ_{a+1} σ_a⁻¹`（Hopf 链环，链接数 = 1）
- **OR(a,b)**：`σ_a⁻¹ σ_{a+1}⁻¹ σ_a`（镜像 Hopf，链接数 = -1）
- **XOR(a,b)**：`σ_a σ_{a+1}² σ_a⁻¹`（Solomon 链环，链接数 = 2）
- **NOT(a)**：`σ_a`（单交叉）

### 数学基础

**辫群 B_n**：n 股绳子的编织操作，生成元 `σ_1, ..., σ_{n-1}`。关系：`σ_i σ_j = σ_j σ_i`（|i-j| ≥ 2），`σ_i σ_{i+1} σ_i = σ_{i+1} σ_i σ_{i+1}`。

**Alexander 定理**：每个纽结/链环都可以表示为闭合辫。

**Reidemeister 变形**：三种局部操作，保持纽结类型不变。用于混淆——每次变形改变图但不改变拓扑。

**链接数**：`lk(C₁, C₂) = (1/2) × Σ(交叉符号)`。拓扑不变量。

**Jones 多项式**：更强的不变量。一般情况计算是 #P-hard。不同纽结可以有相同的 Jones 多项式（如 Kinoshita-Terasaka 和 Conway 纽结）。

**纽结等价**：判断两个纽结图是否表示同一纽结——NP 问题。最佳已知算法有指数级最坏复杂度。

### 攻击结果

| 攻击 | 信息获取 | 结果 |
|---|---|---|
| 图直接分析 | 5% | Reidemeister 变形打乱图 |
| 不变量计算 | 20% | 不变量不唯一 |
| 辫分解 | 40% | 分解不唯一 |
| 纽结等价判定 | 0% | NP 问题 |
| Jones 多项式 | 15% | #P-hard |
| 黑盒差分 | 50% | I/O 已知，结构隐藏 |
| **平均** | **26%** | |

### V8 映射

纽结图是 JSON 对象（交叉点列表）。执行 = 遍历交叉点计算链接数——纯整数运算，O(n)。无浮点、无矩阵、无密码学。纽结图还可视化（可绘制为 ASCII 图或 Canvas 图形）。

### 局限

仅支持 Boolean 电路（AND/OR/NOT/XOR）。无循环、无状态、无字符串操作。适用于小规模高价值函数（License 验证、密码校验），不适用于通用代码保护。

## 信息论 OTP 流保护

### 原理

程序在交付前用一次性密钥（OTP）加密。执行时，客户端逐条从服务器拉取密钥流来解密指令。没有密钥流，代码与随机字节不可区分。

这是唯一达到 **Shannon 完美安全**的方案：`H(P|C) = H(P)`，即密文不含明文的任何信息。

### 架构

```
服务器                              客户端
  │                                   │
  │  K(i) = PRF(MK, session_id, i)    │
  │ ────────────────────────────────→ │ 解密: P[i] = C[i] ⊕ K(i)
  │                                   │ 执行 P[i]
  │  H(K(i)) ←──────────────────────── │ 证明执行了正确指令
  │  验证 H(K(i))                     │
  │                                   │
  │  重复直到程序结束                  │
```

### 数学基础

**Shannon 完美保密**：对于密文 `C = P ⊕ K`，当 K 完全随机且只使用一次时，互信息 `I(P; C) = 0`。攻击者通过分析密文获得的明文信息为零。

**PRF（伪随机函数）**：`K(i) = HMAC-SHA256(master_key, session_id || i)`。每条指令获得唯一的、不可预测的密钥。

**防重放**：每个 index i 在每个会话中只能请求一次。服务器追踪所有已请求的 index。

**前向安全**：每次会话使用不同的 session_id，产生完全不同的密钥流。泄露一个会话不影响其他会话。

### 攻击结果

| 攻击 | 信息获取 | 结果 |
|---|---|---|
| 静态分析 | 0% | Shannon：零信息泄露 |
| 内存转储 | 11% | 每次断点只见 1 条指令 |
| 密钥流截获 | 33% | 必须截获全部密钥（TLS 保护） |
| 重放攻击 | 0% | 会话绑定 + 防重放 |
| 选择明文 | 20% | 仅控制流，不含指令内容 |
| 侧信道 | 10% | 网络抖动掩盖时序 |
| **平均** | **12%** | |

### V8 映射

- 加密代码：Uint8Array（随机字节）
- 密钥下发：WebSocket 或 fetch() 批量预取
- 解密：XOR（V8 原生优化）
- 验证：SHA-256（Web Crypto API）
- 开销：O(n)，每条指令 1 次 XOR + 1 次 HMAC

### 局限

需要网络连接。每条指令增加 1 个 RTT（通过批量预取 4-16 条缓解）。服务器是单点故障——服务器宕机则代码不可执行。这是设计如此。

## 不可区分混淆 (iO)

### 原理

iO 的形式化定义：给定两个功能等价的电路 C₀ 和 C₁，它们的混淆 `iO(C₀)` 和 `iO(C₁)` 在计算意义上不可区分。攻击者即使同时拿到两个混淆版本，也无法判断哪个来自 C₀，哪个来自 C₁。

这不是"难逆向"——而是"证明不可区分"。

### 编译路径

```
经典电路
  → LWE 矩阵编码 (每个门 = LWE 密文)
  → 重新随机化 (LWE 密文重随机化)
  → 密钥切换 (新 LWE 秘密密钥)
  → 噪声淹没 (增大噪声参数)
  → 不可区分的编码电路
```

### 数学基础

**LWE（Learning With Errors）**：给定 `(A, b)`，其中 `b = As + e`（e = 小噪声），区分 s 和随机向量在计算上是困难的。最佳已知攻击是 BKZ 格归约，复杂度 `2^(n/2)`。

**LWE 加密**：密文 `(a, b)`，其中 `b = ⟨a, s⟩ + e + m × ⌊q/2⌋`。m=0 时 b 接近 `⟨a,s⟩`；m=1 时 b 偏移 q/2。

**重随机化**：`(a', b') = (a + r, b + ⟨r, s⟩ + e')`。解密结果不变，但密文看起来完全不同。同一密文的两个重随机化版本在统计上不可区分（在 LWE 假设下）。

### 攻击结果

| 攻击 | 信息获取 | 结果 |
|---|---|---|
| 密钥恢复 | 0% | `q^n = 8192^16` 暴力空间 |
| 不可区分性测试 | 0% | iO 性质成立 |
| 黑盒差分 | 30% | I/O 已知，结构隐藏 |
| 门分析 | 15% | 类型可见，密钥隐藏 |
| 统计推断 | 0% | `1/q^n` 概率 |
| 格归约 (BKZ) | 5% | `2^(n/2) = 2^8 = 256`（n=16 可行，n=256 不可行） |
| **平均** | **8%** | |

### V8 映射

- LWE 密文：Int32Array（矩阵 mod q）
- 执行：矩阵乘法 + 模运算
- 密钥：嵌入客户端或服务器下发
- 开销：`O(n × k)`，n = 电路大小，k = LWE 维度

### 局限

这是教育性实现，n=16（真实安全需要 n ≥ 256）。理论 iO（Diamond iO, 2025）距离实用部署还有数年——执行时间以年计。简化版展示核心思想但不具备密码学安全性。

## 零知识 WASM

### 原理

客户端执行 WASM 字节码，生成零知识证明证明执行正确。服务器验证证明，无需看到代码、输入或执行追踪。**代码从不出现在证明中**。

这是唯一一个代码不在客户端暴露的方案——客户端本地有代码，但它发送给服务器的（也是攻击者能截获的）只是一个证明。

### 架构

```
证明者 (客户端)                    验证者 (服务器)
  │                                  │
  │  执行 WASM → 追踪                 │
  │  追踪 → 算术电路约束               │
  │  约束 + witness → ZK 证明 π       │
  │                                  │
  │  (result, π) ──────────────────→ │  Verify(π)
  │                                  │  通过 / 拒绝
  │                                  │
  │  证明 π 包含:                      │
  │  - 承诺 (哈希)                     │
  │  - 掩码响应                        │
  │  - 公开输入/输出                   │
  │  不包含: 代码、witness             │
```

### 数学基础

**ZK 证明三性质**：

1. **完备性**：正确执行 → 证明总是通过
2. **可靠性**：错误执行 → 证明几乎不可能通过（除可忽略概率外）
3. **零知识**：证明不泄露除"陈述为真"之外的任何信息

**算术电路**：执行追踪编码为 R1CS 约束（`a × b = c mod p`）。每条 WASM 指令生成连接输入输出状态的约束。

**Sigma 协议**：三步交互式证明：

1. 证明者发送承诺 `= H(witness || circuit || nonce)`
2. 验证者发送挑战 `= H(commitment || public_inputs)`——Fiat-Shamir 变换使其变为非交互式
3. 证明者发送响应 `= witness + challenge × blinding (mod prime)`

验证者检查响应但无法得知 witness 的具体值。

### 攻击结果

| 攻击 | 信息获取 | 结果 |
|---|---|---|
| 证明分析 | 0% | 承诺是哈希 |
| Witness 提取 | 0% | 被 blinding factor 掩盖 |
| 证明伪造 | 0% | 需满足所有约束 (NP-hard) |
| 代码恢复 | 0% | 代码不在证明中 |
| 黑盒差分 | 30% | I/O 可观测，代码隐藏 |
| 重放攻击 | 0% | 证明绑定输入 |
| **平均** | **5%** | 六方向最低 |

### V8 映射

- 证明者：WASM 执行 + 证明生成（客户端）
- 验证者：证明验证（服务器）
- 证明格式：JSON（承诺、响应、公开 I/O）
- 开销：O(n) 执行 + O(n) 证明生成

### 局限

ZK-WASM 最适合**可验证计算**（证明你正确执行了代码），而非传统的代码保护（对客户端隐藏代码）。如果攻击者就是客户端本身，代码在本地是可见的。保护针对的是网络观察者和服务器，不是客户端自身。要实现完整保护，需要与 OTP 或 iO 组合使用。

## Local-Mixing 热图消相关

### 原理

将电路的门"打散"到依赖关系允许的最远距离，消除热图相关性。功能相关的门被推到尽可能远的位置，使攻击者无法通过热图分析识别逻辑块。

这是对现有 VMP/WASM 方案最实用的增强——它工作在相同的层面（电路结构），使用简单的整数运算。

### 六种变换

1. **最大距离打散**：在依赖 DAG 允许范围内把门推到最远位置
2. **垃圾门注入**：在真实门之间插入无操作门（10-100x 膨胀）
3. **门分解**：AND → `NOT(NOT(a) OR NOT(b))`（De Morgan，4x 膨胀）
4. **Wire 别名**：插入 identity 门（NOT+NOT）增加数据流复杂度
5. **交换律重排**：随机化 AND/OR/XOR 的输入顺序
6. **等价替换**：用功能等价但表示不同的门序列替换

### 数学基础

**依赖图**：有向无环图 (DAG)，门 = 节点，依赖 = 边。拓扑排序确保执行顺序不被违反。

**热图熵**：门类型在位置上的分布熵。熵越高 = 越均匀 = 越难识别模式。

**聚类分数**：连续同类型门的比例。越低 = 混淆越好。混淆后应接近随机基线。

**消相关验证**：对比混淆电路与原始电路的热图。如果聚类分数下降且平均间隔增大，则消相关成功。

### 攻击结果

|---|---|---|

### V8 映射

- 电路：JSON 数组（门列表，位置已打散）
- 执行：顺序遍历，简单整数运算（AND/OR/XOR/NOT）
- 开销：O(n)，无矩阵、无密码学
- 与现有 VMP 架构最兼容

### 局限

代码仍以可解码形式存在于客户端。如果垃圾门注入不足，模式匹配仍能识别简单电路（如 AND+AND+XOR）。膨胀比（10-100x）会成比例增加执行时间。

## 混合方案建议

### 方案 A：VMP + Local-Mixing（最实用）

```
原始代码 → VMP 字节码 → Local-Mixing 打散 → 混淆 VMP
```

在现有 VMP 基础上增加热图消相关。无额外开销（O(n) + O(n) = O(n)）。防止热图分析识别 VMP handler。现有系统最容易的升级路径。

### 方案 B：OTP + ZK（最安全）

```
原始代码 → OTP 加密 → 客户端解密执行 → ZK 证明 → 服务器验证
```

客户端代码是 Shannon 安全的加密态。执行结果有 ZK 证明（防伪造）。需要网络，适合高价值函数如 License 验证或密钥派生。

### 方案 C：拓扑 + iO（最优雅）

```
原始电路 → 拓扑编码 → iO 混淆 → 双重保护
```

拓扑层提供 NP 硬度。iO 层提供 LWE 硬度。完全离线，无需网络。仅限于小规模 Boolean 电路（License 校验、密码验证）。

### 方案 D：全栈防护

```
              ┌─ ZK 证明 (可验证执行)
              │
原始代码 ──────┼─ OTP 加密 (静态零信息)
              │
              ├─ Local-Mixing (热图消相关)
              │
              └─ VMP 字节码 (基础混淆)
```

四层独立叠加。攻击者需同时破解四层。适合性能次要、安全优先的军事级保护。

## 局限与未来方向

### 各方案关键局限

| 方案 | 关键局限 |
|---|---|
| 量子混淆 | 经典 CPU 上 2^n 指数开销；仅概念性 |
| 拓扑保护 | 仅 Boolean 电路；无循环/状态/字符串 |
| OTP 流 | 需要网络；服务器是单点故障 |
| iO | 教育性 n=16；真实安全需 n≥256；距实用数年 |
| ZK-WASM | 保护网络观察者，不保护客户端自身 |
| Local-Mixing | 代码仍可解码；模式匹配可识别简单电路 |

### 未来方向

1. **量子**：探索量子启发的经典算法（张量网络），在无指数开销的情况下近似量子行为
2. **拓扑**：通过持久同调扩展到 Boolean 电路之外，支持循环和状态
3. **OTP**：实现 WebSocket 密钥流 + 自适应预取以最小化延迟
4. **iO**：集成真实 FHE 库（SEAL、OpenFHE）支持非线性门
5. **ZK-WASM**：使用生产级 ZK-SNARK 库（zkSync、Polygon Miden）
6. **Local-Mixing**：添加动态垃圾生成（每次执行垃圾不同）防止跨运行对比

### 开放问题

- 这些方案能否在不产生指数开销的情况下组合？
- 哪种方案对 AI 辅助逆向（LLM 去混淆）抵抗力最强？
- 与硬件保护（TEE、SGX）在浏览器上下文相比如何？
- 可证明安全的代码混淆是否存在根本性的开销下界？

## 结论

传统 JSVMP 及其变体都共享一个根本假设：程序以固定的、可复制的形式存在于客户端。本文探索的 6 个方向以不同方式打破了这个假设——通过量子力学、拓扑学、信息论、格密码学、零知识证明和热图消相关。

每个方案都有明显的权衡：

- **ZK-WASM** 信息泄露最低（5%），但最适合可验证计算而非代码隐藏
- **OTP 流** 达到 Shannon 完美静态安全，但需要网络
- **iO** 提供可证明的不可区分性，但距实用部署数年
- **拓扑保护** 提供 NP 硬度和 O(n) 执行，但仅限 Boolean 电路
- **Local-Mixing** 是最实用的 VMP 增强，无密码学开销
- **量子混淆** 展示了物理定律提供的根本不同的保护模型

对于今天的实际部署，推荐路径是 **在现有 VMP/WASM 上叠加 Local-Mixing**，对高价值函数使用 **OTP 流**。两者都使用 V8 原生优化的简单运算，无指数开销，可以立即落地。
